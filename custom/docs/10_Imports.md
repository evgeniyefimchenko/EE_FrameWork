# Импорт типов, свойств и наборов из JSON

Этот раздел описывает JSON-формат импорта для трёх сущностей:

- типов свойств;
- свойств;
- наборов свойств.

Импорт не создаёт:

- категории;
- типы категорий;
- страницы;
- реальные значения свойств у сущностей.

Он работает только со структурой property layer.

## Общая идея

Импорт работает как `upsert` по практическим ключам приложения:

- тип свойства ищется по `name + language_code`;
- свойство — по `name + type + language_code`;
- набор свойств — по `name + language_code`.

Поле `code` в JSON используется только как внутренняя ссылка внутри файла.

В БД оно не сохраняется.

## Корневой объект

```json
{
  "schema": "ee_property_definitions_import",
  "version": 1,
  "language_code": "RU",
  "property_types": [],
  "properties": [],
  "property_sets": []
}
```

Обязательные поля:

- `schema`
- `version`

Необязательные:

- `language_code`
- `property_types`
- `properties`
- `property_sets`

## Типы свойств

Пример:

```json
{
  "code": "seo_text_block",
  "name": "SEO текстовые поля",
  "status": "active",
  "description": "Текстовые SEO-поля",
  "fields": ["text", "textarea"]
}
```

Поля:

- `code` — внутренний идентификатор внутри JSON
- `name` — имя типа свойства
- `status` — `active`, `hidden`, `disabled`
- `description` — описание
- `fields` — обязательный список типов полей

Поддерживаемые типы `fields`:

- `text`
- `number`
- `date`
- `time`
- `datetime-local`
- `hidden`
- `password`
- `file`
- `email`
- `phone`
- `select`
- `textarea`
- `image`
- `checkbox`
- `radio`

## Свойства

Пример:

```json
{
  "code": "seo_meta",
  "type_code": "seo_text_block",
  "name": "SEO мета",
  "status": "active",
  "sort": 100,
  "entity_type": "page",
  "is_multiple": 0,
  "is_required": 0,
  "description": "SEO-блок страницы",
  "default_values": []
}
```

Поля:

- `code`
- `type_code`
- `name`
- `status`
- `sort`
- `entity_type`
- `is_multiple`
- `is_required`
- `description`
- `default_values`

`entity_type` допускает:

- `category`
- `page`
- `all`

## Главное правило `default_values`

`default_values` должен:

- быть массивом;
- иметь ту же длину, что и `fields` связанного типа;
- сохранять тот же порядок, что и `fields`.

Если тип состоит только из простых полей вроде `text`, `textarea`, `number`, `date`, `email`, `phone`, `file`, `image`, минимальная структура может собраться автоматически.

Если есть:

- `select`
- `checkbox`
- `radio`

описание `default_values` обязательно.

## Простые поля

Пример:

```json
{
  "type": "text",
  "label": "Meta title",
  "default": "",
  "required": 0,
  "multiple": 0
}
```

Допустимые ключи:

- `type`
- `label`
- `title`
- `default`
- `required`
- `multiple`

## Поля выбора

Для `select`, `checkbox` и `radio` структура должна описывать не только label, но и набор вариантов.

Практически важно следить за тремя вещами:

- набор `options` должен быть однозначным;
- `default` должен быть подмножеством реально объявленных ключей;
- `multiple` должен соответствовать реальной логике поля.

Для `select`, `checkbox` и `radio` вариантам можно задавать явный `value`.

Это особенно важно при миграции из внешних систем, если исходные данные уже хранятся в кодах вроде:

- `1`
- `2`
- `100`
- `99`

Тогда импорт сохраняет эти коды как реальные ключи вариантов, а не придумывает их заново по индексу или slug из label.

## Наборы свойств

Набор описывает только состав свойств.

Если набор уже существует, его состав после импорта становится равен составу, описанному в JSON.

Это значит, что импорт набора — не “добавление ещё одного свойства”, а синхронизация целевой структуры.

## Что импорт валидирует жёстко

Импорт не должен пропускать:

- битый JSON;
- дубли `code`;
- неизвестный `type_code`;
- несовпадение длины `fields` и `default_values`;
- неподдерживаемые типы полей;
- внутренне противоречивые choice-структуры.

Именно такие ошибки должны блокировать просмотр или подтверждение импорта.

## Что важно помнить перед импортом

- импорт меняет структуру, а не только “добавляет новые записи”;
- изменение типа свойства и состава наборов может запустить lifecycle;
- после крупных структурных изменений нужно быть готовым к пересборке зависимых значений.

## Где использовать этот формат

Формат подходит для:

- переноса property-структуры между инсталляциями;
- пакетной загрузки типов/свойств/наборов;
- поддерживаемого структурного импорта, который повторяет реальную модель EE_FrameWork.

В админке этот сценарий доступен через импорт типов, свойств и наборов. Если вы редактируете формат импорта, держите этот документ синхронным с реальным поведением import-кода.

## Плановый запуск профилей импорта

Если речь идёт не о JSON-импорте структуры свойств, а о профилях WordPress-импорта в админке, для них теперь рекомендуется отдельная схема:

- серверный cron запускает только `php app/cron/run.php`;
- сам профиль импорта связывается с cron-агентом через handler `import.profile`;
- в payload хранится `job_id` профиля;
- управлять расписанием и запуском можно из админки, не плодя отдельные cron-команды под каждый профиль.

## Миграционный контракт URL и donor-ссылок

WordPress-профиль импорта теперь умеет работать в двух связанных режимах:

- `Сохранять исходные WP пути как public URL (route_path)`
- `Переписывать ссылки сайта-донора на локальный домен во время импорта`

Как это работает:

- экспортёр кладёт в пакет не только `slug`, но и `source_path`;
- импортёр может сохранить этот `source_path` в core-поле `route_path`;
- public router использует `route_path` раньше обычной slug-иерархии;
- абсолютные ссылки сайта-донора в описаниях и rich-text полях автоматически переписываются на локальный домен;
- если ссылка указывает на уже импортированную категорию или страницу, берётся её текущий public URL;
- если сущность не была импортирована, сохраняется исходный path уже на локальном домене.

Это нужно для сценария полного переезда, когда важно сохранить старую ссылочную структуру:

```text
https://donor.example/resort/catalog/flat/
https://project.example/resort/catalog/flat/
```

`route_path` опционален. Если он не используется, система продолжает строить public URL по обычной EE-модели из `slug`.

## Фоновая догрузка медиа после импорта

После контентного импорта внешние изображения и файлы попадают в таблицу очереди `ee_import_media_queue`.
Дальше их порциями обрабатывает системный агент `media-mirror-worker`.

Статусы очереди:

- `queued` — файл ждёт первой или повторной обработки;
- `running` — файл сейчас обрабатывает worker;
- `failed` — временная ошибка, элемент будет повторён позже;
- `terminal_failed` — без повтора. Такой статус ставится для безвозвратно битых источников, например `HTTP 404/410`;
- `done` — файл успешно догружен, сохранён локально и перепривязан в данных.

То есть production-путь для повторяющегося импорта должен идти через cron-агенты, а не через набор ручных scheduler-строк.

## WordPress-миграция: практический режим

Для миграции курортов и объектов из WordPress теперь используется такой базовый подход:

- `taxonomy:resort` импортируется как категории;
- `post_type:object` импортируется как страницы;
- пользователи импортируются отдельно, но не получают старый WordPress-пароль как рабочий пароль EE_FrameWork;
- связь “объект -> владелец” должна ехать из экспортного пакета явно, а не только через текстовые meta-поля.

### Что изменено в экспортёре

Экспортёр WordPress теперь добавляет в `pages.jsonl` поля владельца:

- `owner_user_source_id`
- `owner_user_login`
- `owner_user_email`
- `owner_user_name`

### Фоновая догрузка медиа после импорта

Импорт структуры и контента теперь не должен висеть на скачивании всех картинок сразу.

Правильный production-сценарий такой:

- основной импорт создаёт категории, страницы, пользователей и значения свойств;
- после `finalize` импорт автоматически собирает внешние URL из:
  - `description`
  - `short_description`
  - `image/file` payload внутри `property_values`
- найденные медиа попадают в таблицу очереди `ee_import_media_queue`;
- дальше их обрабатывает встроенный системный cron-агент `media-mirror-worker`.

Важно:

- агент создаётся автоматически при установке системы;
- он постоянно включён;
- серверу по-прежнему нужен только один минутный cron:

```bash
php /var/www/html/app/cron/run.php
```

Это значит, что импортёр не плодит временных агентов под каждый пакет. Он только кормит очередь, а системный worker спокойно догружает медиа порциями, не перегружая сервер.

Это нужно для восстановления связи страницы с пользователем внутри EE_FrameWork без догадок по имени.

### Что изменено в импортёре

При импорте WordPress-пользователей система:

- создаёт или обновляет пользователя;
- помечает его как требующего назначения нового пароля;
- сохраняет migration-метаданные в `options`;
- не использует WordPress hash как готовый пароль для нового auth-контра EE_FrameWork.

Практически это означает:

- после импорта пользователь должен сам назначить новый пароль при первом входе;
- признак задаётся через `options.auth.require_password_setup`;
- причина помечается как `wp_migration`.

### Curated schema для проекта

Для производственной миграции рекомендуется готовить отдельный curated schema JSON вне репозитория фреймворка.

В таком файле обычно фиксируются:

- курорты как свойства категорий;
- объекты размещения как свойства страниц;
- составное свойство `Номера и цены`;
- до 20 ценовых периодов внутри одного номера;
- реальные коды вариантов WordPress для `select`, `checkbox` и `radio`.

Такой файл относится к конкретному проекту и не должен жить в исходном репозитории EE_FrameWork.

### Что важно при следующей выгрузке

Для следующего WP-экспорта рекомендуется:

- оставить только нужные источники `taxonomy:resort` и `post_type:object`;
- выгружать `ACF = all`, если проект реально использует ACF;
- не включать служебные private meta-ключи (`_...`), если нет отдельной задачи по их переносу.

Иначе в пакет снова приедет большой объём технического шума, который не нужен для curated-миграции.

## Жёсткая политика по private ACF meta

Для WordPress-импорта в EE_FrameWork теперь действует жёсткое правило:

- `_thumbnail_id` разрешён как служебное исключение для картинок;
- `field_*` и `_field_*` считаются чисто техническими ACF-ключами и не импортируются;
- private ACF reference meta вроде `_photos`, `_map`, `_address`, если у них есть публичная пара, не импортируются;
- служебные WordPress meta вроде `_edit_lock`, `_edit_last`, `_wp_old_slug`, `_wp_attachment_*`, `_oembed_*`, `_menu_item_*` тоже не импортируются.

Это сделано затем, чтобы curated-импорт не засорял property layer техническими ссылками ACF и внутренними хвостами WordPress.

## Mixed content в описаниях

Контент `post_content` и текстовые блоки WordPress неоднотипны:

- часть записей уже содержит HTML;
- часть хранится как plain text с пустыми строками;
- часть приходит в mixed формате: текст + отдельные HTML-блоки.

Поэтому импорт использует двухступенчатую нормализацию:

- экспортёр старается привести rich text к более нормальному HTML ещё на стороне WordPress;
- импортёр дополнительно превращает plain text и mixed content в безопасный HTML с абзацами и переносами строк.

Практический смысл этого правила:

- администратор не должен получать после импорта простыни сырого текста вместо абзацев;
- чистый повторный прогон должен давать уже пригодное для фронта описание без ручного переписывания контента.

## Автоматические lifecycle mapping для WP

Для объектных страниц WordPress-импорт дополнительно знает два безопасных mapping-а в внутренний lifecycle-набор:

- `postmeta:zametki` -> `Внутренний комментарий менеджера`
- `postmeta:published_to` -> `Размещение активно до`

`published_to` при импорте нормализуется в ISO-дату `YYYY-MM-DD`.

Остальные бизнес-ключи вроде:

- `has_payment`
- `how_publish`
- `last_payment`
- `views_count`

пока не маппятся автоматически в curated property layer.

Причина простая: часть из них относится к будущему billing/statistics-контуру, а часть требует отдельного продуктового решения, а не “догадки во время импорта”.
