# Cron-агенты и scheduler

В EE_FrameWork периодические фоновые задачи устроены как агентный scheduler.

Идея простая:

- в системный cron ставится только один минутный запуск;
- код обработчиков хранится в проекте;
- расписание, payload и состояние задач лежат в БД;
- админка управляет агентами без правки crontab.

## Базовая схема

```text
system cron -> php app/cron/run.php
            -> inc/cli.php cron:run-agents
            -> CronAgentService
            -> due agents from DB
            -> CronAgentRegistry handlers
```

Сервер знает только одну боевую команду:

```bash
php app/cron/run.php
```

А уже внутри приложения решается:

- какой агент пора запускать;
- можно ли его запускать сейчас;
- не перегружен ли сервер;
- не завис ли прошлый запуск.

## Где что хранится

### В файлах проекта

В коде хранятся:

- `app/cron/run.php` — минутный wrapper для scheduler-а;
- `inc/cli.php` — общий CLI entrypoint;
- `classes/system/CronAgentService.php` — планирование, блокировки, лимиты, история;
- `classes/system/CronAgentRegistry.php` — встроенные handler-ы;
- админский UI управления агентами.

### В базе данных

В БД лежат:

- `ee_cron_agents` — описание агентов;
- `ee_cron_agent_runs` — история запусков.

История запусков не должна расти бесконечно. Scheduler автоматически чистит `ee_cron_agent_runs`:

- по сроку хранения;
- и по верхнему лимиту строк.

Это защищает БД от бесконтрольного роста истории при минутном cron.

В БД не должно храниться PHP-кода обработчика. Правильная модель такая:

- в БД лежит `handler = import.profile`;
- в коде живёт реализация, что именно делает `import.profile`.

## Что хранит один агент

Обычно у агента есть:

- `code` — стабильный код задачи;
- `handler` — имя обработчика;
- `schedule_mode` — `interval | cron | manual`;
- `interval_minutes` или `cron_expression`;
- `payload_json` — параметры запуска;
- `is_active`;
- `priority`;
- `weight`;
- `max_runtime_sec`;
- `lock_ttl_sec`;
- `retry_delay_sec`;
- `next_run_at`;
- lock-поля и last-run поля.

## Почему это лучше, чем россыпь cron-файлов

Такой подход решает сразу несколько проблем:

- на сервере не разрастается crontab;
- включение и отключение задач делается из админки;
- одно и то же задание не должно стартовать дважды одновременно;
- появляется история запусков;
- можно видеть ошибки, зависания и следующее время запуска.

## Защита от перегрузки

В scheduler заложено несколько уровней защиты.

### 1. Глобальный lock scheduler-а

Одновременно только один процесс должен считаться главным минутным тикером.

### 2. Lock на конкретный агент

У каждого агента есть свои поля блокировки:

- `locked_at`
- `locked_until`
- `locked_by`
- `current_run_token`

Это не позволяет двум процессам взять один и тот же агент.

### 3. Лимиты нагрузки

Scheduler учитывает:

- лимит задач за тик;
- суммарный вес задач за тик;
- лимит одновременного выполнения.

Это позволяет запускать тяжёлые задачи осторожно и не перегружать сервер.

### 4. Защита от overlap и OOM

На практике одной только DB-блокировки scheduler-а недостаточно.

Если минутный cron стартует новый `php app/cron/run.php`, пока прошлый процесс ещё не вышел, сервер может получить каскад из нескольких CLI-процессов и уткнуться в память ещё до того, как scheduler успеет корректно отказаться от второго запуска.

Поэтому в `app/cron/run.php` используется ранний файловый lock:

- новый тик не входит глубоко в bootstrap, если прошлый ещё жив;
- это защищает от наложения минутных запусков;
- защита работает даже если БД временно недоступна.

Отдельно действуют memory guard-ы:

- `ENV_MEMORY_LIMIT_CLI`
- `ENV_CRON_MEMORY_SOFT_LIMIT_MB`
- `ENV_CRON_AGENT_MEMORY_SOFT_LIMIT_MB`
- `ENV_MEDIA_MIRROR_MEMORY_SOFT_LIMIT_MB`

Жёсткий лимит задаёт `memory_limit`, а мягкие лимиты нужны для graceful stop:

- scheduler завершает тик до OOM;
- `media.mirror.worker` останавливает batch и продолжает на следующем тике;
- MariaDB и web runtime не должны становиться жертвами одного тяжёлого cron-процесса.

### 5. Восстановление stale lock-ов

Если процесс умер посередине выполнения, агент не должен зависнуть навсегда в статусе `running`.

Для этого используются:

- `locked_until`;
- recovery stale locks;
- история запусков с failed-завершением stale run-ов.

Отдельно на `/admin/health` теперь есть единая operational recovery-кнопка:

- она одним действием проверяет `cron`, `property lifecycle`, `media queue` и `backup queue`;
- stale состояния переводятся обратно в контролируемый режим;
- сам health-screen сразу показывает, что именно требует внимания.

## Встроенные handler-ы

Сейчас в ядре есть handler-ы:

- `property_lifecycle.next`
- `search.popularity.update`
- `import.profile`
- `media.mirror.worker`
- `ops.health_check`

Это не значит, что все они обязаны быть активны по умолчанию.

Важно:

- обязательные системные агенты, нужные для работы платформы, создаются автоматически при развёртывании и первом bootstrap;
- опциональные агенты вроде `import.profile` создаются по месту использования или через админку.

## Постоянный media worker

Для миграций WordPress и других внешних источников в системе есть постоянный системный агент:

- `code = media-mirror-worker`
- `handler = media.mirror.worker`

Он создаётся автоматически при install/bootstrap и всегда остаётся включённым.

Его задача:

- брать элементы из `ee_import_media_queue`;
- скачивать внешние файлы в локальное хранилище;
- регистрировать их в `ee_files`;
- переписывать импортированные значения свойств и HTML-поля с внешних URL на локальные.

Это важный принцип:

- импорт не создаёт отдельного агента под каждый запуск;
- импорт только пополняет очередь;
- системный агент сам подхватывает задания и работает порциями по расписанию.

Важно понимать статусы media queue:

- `failed` — временная ошибка, элемент будет повторён;
- `terminal_failed` — источник подтверждённо битый, повторять его бессмысленно;
- `running + locked_until < NOW()` — stale lock, который восстанавливается отдельно и также подсвечивается на health-screen.

## Как это настраивается в админке

Главный экран:

```text
/admin/cron_agents
```

Там можно:

- увидеть сводку;
- создать нового агента;
- включить или выключить агент;
- запустить его вручную;
- открыть историю запусков;
- восстановить stale locks.

Редактирование агента:

```text
/admin/cron_agent_edit/id/<agent_id>
```

На экране редактирования есть два уровня настройки:

- удобные поля `Параметры handler-а` для типовых handler-ов;
- raw `Payload JSON` для расширенных сценариев и тонкой ручной настройки.

История запусков:

```text
/admin/cron_agent_runs
```

## Как лучше настраивать импорт

Для профилей импорта лучший путь такой:

1. настроить профиль импорта;
2. создать cron-агент с handler `import.profile`;
3. положить в `payload_json`:

```json
{
  "job_id": 15
}
```

4. задать расписание;
5. оставить системный cron на минутный запуск `app/cron/run.php`.

Это лучше, чем плодить отдельные cron-команды под каждый профиль.

В админке профиль импорта можно связать с агентом прямо с экрана редактирования профиля: система умеет создавать или синхронизировать связанный `import.profile` агент без ручного заполнения payload.

## Практика для media-mirror worker

У системного `media-mirror-worker` два самых важных параметра:

- `batch_limit` — сколько файлов обрабатывать за один минутный тик;
- `retry_delay_sec` — через сколько секунд повторять неудачную догрузку файла.

Если источник внешнего файла уже безвозвратно битый, worker не должен гонять его бесконечно.
Для этого очередь медиа использует отдельный статус `terminal_failed`.
Сейчас в него переводятся, например, ответы `HTTP 404/410`, и такие элементы больше не возвращаются в работу автоматически.

Эти параметры доступны прямо в админке на экране редактирования cron-агента и синхронизируются с `payload_json`.

Для повседневной эксплуатации не обязательно открывать полное редактирование агента:

- на `/admin/cron_agents` есть отдельная быстрая панель `media-mirror-worker`;
- там видно размер очереди;
- там же можно сразу менять `batch_limit` и `retry_delay_sec`;
- и вручную запустить worker одним кликом.

## Когда использовать manual mode

`manual` подходит для задач, которые:

- не должны стартовать автоматически;
- запускаются только из админки;
- нужны как ручной operational инструмент.

## Типовые проблемы

### Агент не запускается

Проверьте:

- включён ли он;
- не `manual` ли это агент;
- не стоит ли `next_run_at` в будущем;
- не блокирует ли его старый lock;
- не упирается ли scheduler в лимиты нагрузки.

### Агент всё время due, но не исполняется

Смотрите:

- историю запусков;
- канал `cron_agents` в логах;
- stale locks;
- правильность `handler` и `payload_json`.

### Сервер перегружается

Уменьшайте:

- `max_agents_per_tick`;
- `max_weight_per_tick`;
- `max_concurrent`;
- либо повышайте `weight` тяжёлых задач, чтобы scheduler брал их осторожнее.

## Практический production минимум

Для production достаточно такой схемы:

```cron
* * * * * php /var/www/html/app/cron/run.php >/dev/null 2>&1
```

Дальше вся работа уже идёт через:

- таблицы cron-агентов;
- очередь `ee_import_media_queue`;
- админский UI;
- встроенные ограничения scheduler-а.
