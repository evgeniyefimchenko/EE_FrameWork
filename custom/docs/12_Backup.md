# Резервное копирование

## Что реализовано в EE_FrameWork

Резервное копирование в EE_FrameWork больше не выполняется тяжёлым синхронным web-запросом. Подсистема работает через:

- отдельный экран `/admin/backup`;
- планы резервного копирования в БД;
- очередь backup-задач в БД;
- системный агент `backup-queue-worker`;
- локальные снапшоты в `tmp/backups/`;
- удалённые профили хранения через `SFTP` и `FTP`.

Это означает:

- администратор ставит backup в очередь;
- минутный scheduler запускает системного worker-а;
- worker создаёт локальный снапшот;
- при необходимости worker отправляет архивы на удалённый хост.

## Планы резервного копирования

Вместо одного жёстко зашитого сценария администратор теперь создаёт несколько backup-планов. Каждый план хранит:

- код и название;
- описание;
- состав БД;
- состав файловой части;
- способ доставки;
- целевой удалённый профиль;
- флаги `active` и `default`.

### Состав базы данных

Для БД доступны режимы:

- `Все таблицы`
- `Только выбранные таблицы`
- `Все таблицы кроме выбранных`
- `Без базы данных`

Список таблиц для конкретного плана берётся из текущей БД и сохраняется прямо в план.

### Состав файловой части

Для файлов доступны режимы:

- `Все доступные файлы и папки`
- `Только выбранные файлы и папки`
- `Все файлы и папки кроме выбранных`
- `Без файловой части`

Сейчас в каталог выбора входят:

- `app/`
- `assets/`
- `classes/`
- `layouts/`
- `custom/`
- `uploads/`
- `logs/`
- `cache/`
- `.htaccess`
- `index.php`
- `error.php`
- `inc/configuration.php`

Это позволяет, например, держать такие практичные планы:

- `Только БД`
- `Рекомендуемый локальный backup`
- `Полный snapshot без cache/logs`
- `Только uploads + БД`
- `Только выбранные таблицы и custom/`

### Что входит в локальный snapshot

Worker не придумывает состав на лету. Он берёт уже подготовленный план, а в очередь backup-задачи записывается снимок этого состава:

- точный список таблиц БД на момент постановки в очередь;
- точный список файлов/папок на момент постановки в очередь;
- режим доставки и удалённый профиль.

Это важно, потому что план может измениться позже, а уже queued-задача должна остаться предсказуемой.

## Как это запускается

На сервере нужен один системный cron:

```bash
php app/cron/run.php
```

Он уже используется всей агентной системой, поэтому backup не требует отдельного cron-файла.

## Удалённые хранилища

В админке можно завести профили удалённого хранилища:

- `SFTP`
- `FTP`

Профиль удалённого хранилища хранит:

- код;
- имя;
- протокол;
- host и port;
- логин;
- пароль;
- удалённый путь;
- timeout;
- для FTP — passive mode;
- флаги `active` и `default`.

Режим доставки теперь задаётся на уровне плана:

- `Только локально`
- `Локально и на удалённый хост`
- `Удалённый хост обязателен`

Если план ссылается на конкретный профиль, будет использован он. Если план хранит `target_id = 0`, система использует профиль по умолчанию.

## Статусы backup-задач

Очередь хранит историю и текущее состояние:

- `queued`
- `running`
- `done`
- `partial`
- `failed`

`partial` означает, что локальный snapshot создан, но удалённая отправка завершилась с ошибкой.

Если backup-задача зависла в `running`, это теперь видно на `/admin/health`:

- health-screen показывает количество stale backup jobs;
- единая кнопка recovery проверяет и backup-контур тоже;
- после recovery stale-задача переводится в контролируемое failed-состояние и не остаётся висеть бесконечно.

## Retention

Локальные snapshot-директории не растут бесконечно. Подсистема применяет retention-политику:

- `ENV_BACKUP_RETENTION_DAYS`
- `ENV_BACKUP_MAX_LOCAL_SNAPSHOTS`

По умолчанию:

- хранить `14` дней;
- держать не больше `20` локальных снапшотов.

Сами планы резервного копирования не одноразовые: их можно переиспользовать для ручного запуска из `/admin/backup`, а backup-worker просто берёт queued-задачи и исполняет их в фоне.

## Что ещё не делает backup-подсистема

На текущем этапе backup не решает всё сразу. Пока не реализованы:

- restore UI;
- инкрементальные backup-цепочки;
- шифрование архивов на удалённом хосте;
- remote retention policy;
- offsite verify/restore-check.

Но уже сейчас это полноценный рабочий production-контур:

- запуск идёт в фоне;
- есть история задач;
- есть удалённые профили;
- есть offsite upload;
- есть защита от разрастания локальных snapshot-ов.
