# Архитектура и поток запроса

EE_FrameWork использует прямой и понятный поток обработки:

```text
URL -> Router -> Controller -> Model -> View -> Layout -> Response
```

## Слои и ответственность

## Router

Отвечает за разбор URL и выбор:

- файла контроллера;
- имени контроллера;
- action;
- массива аргументов.

Router не должен выполнять бизнес-логику. Его задача — только безопасно и предсказуемо довести запрос до контроллера.

## Controller

Контроллер управляет сценарием:

- проверяет доступ;
- грузит модели;
- читает входные данные;
- вызывает mutation/read операции;
- выбирает view и layout;
- задаёт redirect или response.

Контроллер не должен:

- писать SQL руками в нескольких местах;
- хранить сложную бизнес-логику;
- смешивать сохранение данных с разметкой.

## Model

Модель работает с данными и инвариантами:

- SQL-запросы;
- правила сохранения;
- транзакции;
- lifecycle-операции;
- формирование `OperationResult` для мутаций.

Граница стандарта здесь важна:

- чтение может возвращать массивы и scalar-значения;
- мутации наружу должны возвращать `OperationResult`.

## View

`View` отвечает за рендер шаблонов и передачу данных в представления.

Это слой:

- подстановки переменных;
- чтения шаблонов;
- частичного block-cache;
- рендера динамических блоков.

## Layout

Layout собирает итоговую страницу:

- `<head>`;
- общие стили и скрипты;
- общий каркас страницы;
- `layout_content`, который подготовил контроллер.

## Где хранится состояние

### Сессии

Сессия хранит короткоживущий runtime-контекст:

- язык интерфейса;
- auth/session state;
- некоторые redirect и error-cases.

### Пользовательские опции

Пользовательские UI-настройки лежат в `ee_users_data.options`.

Используйте это для:

- локали;
- UI-настроек;
- несекьюрных флагов интерфейса.

Не используйте это как единственный источник истины для security-критичных решений.

### Кэш

Система хранит несколько типов кэша:

- route cache;
- HTML cache;
- block cache.

Ключевой принцип:

- кэш — ускорение чтения;
- источник истины — БД и runtime-код.

## Точки расширения

### configuration.php vs bootstrap.php

- `inc/configuration.php` — статический слой настроек.
- `inc/bootstrap.php` — ранняя runtime-подготовка: вычисляемые URL/paths, define-констант, shutdown/fatal hooks, canonical redirect policy, подключение `startup.php`.

Если в проекте появляется новая функция, helper или runtime-policy, ей не место в `configuration.php`.

### custom/

`custom/` — единственная upgrade-safe точка проектного кода.

Используйте:

- `custom/hooks.php`
- `custom/bootstrap.php`
- `custom/src/...`

### Hook API

Через `Hook::add/run/filter/until` можно:

- подписаться на core-события;
- добавить post-processing;
- изменить данные до/после операции;
- строить integration layer без правок ядра.

### Helper и service-классы

Если логика не является маршрутом, не является view и не обязана жить в ядре, выносите её в сервисный класс.

Типичный пример:

- `custom/src/OrderStatusSync.php`
- `custom/src/MarketplaceWebhookHandler.php`

## Что считается хорошим архитектурным стилем в EE_FrameWork

- маршруты и UI-модули живут в `app/`;
- project-specific поведение живёт в `custom/`;
- мутации моделей возвращают `OperationResult`;
- hooks используются для расширения, а не для сокрытия плохой архитектуры;
- layout и view не принимают решения о данных;
- кэш не маскирует проблемы данных и маршрутизации.
